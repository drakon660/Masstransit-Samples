# MassTransit Schedule in a State Machine Saga

Notes on how `Schedule<TSaga, TMessage>` works inside a `MassTransitStateMachine`, based on the `ReservationStateMachine` in this sample.

## The declaration

```csharp
public Schedule<Reservation, ReservationExpired> ExpirationSchedule { get; set; }
```

Declaring the property alone is **not** enough — MassTransit validates the state machine at startup and throws `ConfigurationException: ExpirationSchedule.AnyReceived was not specified` until the schedule is both configured and wired.

## Minimum wiring required

```csharp
Schedule(() => ExpirationSchedule, x => x.ScheduleId, x =>
{
    x.Delay = TimeSpan.FromHours(24);
    x.Received = r => r.CorrelateById(m => m.Message.ReservationId);
});
```

Three inputs, three jobs:

| Argument | Purpose |
|---|---|
| `() => ExpirationSchedule` | The schedule property on the state machine. |
| `x => x.ScheduleId` | Saga property that holds the current scheduled-message **tokenId**. Used to detect stale/cancelled schedules. Nullable `Guid?`. |
| `x.Delay` | Default delay applied when `context.Schedule(ExpirationSchedule, ...)` is called inside a behavior. |
| `x.Received = r => r.CorrelateById(...)` | How to locate the saga instance when the scheduled message comes back — same role as `Event(() => X, x => x.CorrelateById(...))` for normal events. **This is what satisfies the `AnyReceived was not specified` validation.** |

> If you also declare `Event(() => ReservationExpired, x => x.CorrelateById(...))` for the same message type, it's **redundant and can conflict** — the `Schedule<,>` already registers `ExpirationSchedule.Received` as the event. Use `ExpirationSchedule.Received` / `ExpirationSchedule.AnyReceived` in behaviors instead.

## The two ids — and why both are needed

A schedule uses two independent ids to coordinate:

| Id | Stored where | Purpose |
|---|---|---|
| **Business correlation id** (e.g. `ReservationId`) | On the message payload + matches `saga.CorrelationId` | "**Which saga instance** does this message belong to?" Used by `x.Received = r => r.CorrelateById(...)` to find the saga row. |
| **Schedule tokenId** (Guid, generated by MassTransit) | Message header + saga property pointed to by `x => x.ScheduleId` | "**Is this scheduled message still valid** for that saga, or is it stale?" Used after the saga is loaded, to decide `Received` vs `AnyReceived`. |

### Why the business id alone is not enough

1. Saga schedules expiration → `ScheduleId = tokenA`.
2. Something cancels or reschedules → `ScheduleId = tokenB` (or `null`).
3. `tokenA`'s fire time arrives anyway (Quartz already had it queued).
4. Message lands carrying `ReservationId = 123` and header `TokenId = tokenA`.
   - Business id lookup → finds the saga.
   - Token compare → `tokenA ≠ tokenB` → MassTransit knows it's stale, skips `Received`, only fires `AnyReceived` (usually a no-op).

Without the token, step 3 would re-run the expiration logic even though you already cancelled it. The two-id design prevents that race.

## End-to-end flow

1. **Schedule action runs** — `context.Schedule(ExpirationSchedule, ...)` inside a behavior. MassTransit generates a **tokenId (Guid)** client-side, attaches it as a header (`MT-Scheduling-TokenId`), and sends a `ScheduleMessage` command to the scheduler queue (e.g. `queue:quartz`).
2. **Scheduler persists it** — Quartz writes the job to SQL with the fire time. The tokenId travels along.
3. **MassTransit saves the tokenId** into the saga property named in `x => x.ScheduleId` so the saga remembers "this is my currently-pending schedule."
4. **Time elapses → scheduler fires** — Quartz republishes the original message (`ReservationExpired`) back onto the bus with that same tokenId header.
5. **Saga receives it:**
   - `x.Received = r => r.CorrelateById(...)` locates the saga.
   - MassTransit compares the header tokenId against `saga.ScheduleId`:
     - match → fires `ExpirationSchedule.Received` (fresh schedule).
     - mismatch / null → fires `ExpirationSchedule.AnyReceived` only (stale — e.g. cancelled or rescheduled since).

## `Received` vs `AnyReceived`

- `ExpirationSchedule.Received` — scheduled event fired when the incoming tokenId matches `saga.ScheduleId`. This is the normal path; put your expiration logic here.
- `ExpirationSchedule.AnyReceived` — fires regardless of token match. Useful as a safety net, or left as a no-op finalizer. MassTransit requires the schedule's correlation (`x.Received = ...`) to be specified either way — that's what the `AnyReceived was not specified` validation is actually checking.

## `Schedule<TSaga>` vs `Schedule<TSaga, TMessage>`

`Schedule<TSaga>` is the **non-generic base interface** that `Schedule<TSaga, TMessage>` implements. You'd rarely declare it directly on a state machine — its main uses are:

### 1. Polymorphic / message-agnostic operations

The base interface exposes operations that don't need the message type:

- `Unschedule(context)` — cancel the pending schedule
- Access to the tokenId property / schedule name

So generic helper code that just needs to cancel *some* schedule without caring what message type it carries can accept `Schedule<TSaga>`:

```csharp
static async Task CancelIfPending<TSaga>(BehaviorContext<TSaga> ctx, Schedule<TSaga> schedule)
    where TSaga : class, SagaStateMachineInstance
{
    await schedule.Unschedule(ctx);
}
```

### 2. When you don't need `Received` / `AnyReceived` events

`Schedule<TSaga, TMessage>` adds the event members used for handling the scheduled message when it fires. If a schedule is **fire-and-something-else-handles-it** (e.g. the scheduler publishes a message that a different consumer processes, not the saga itself), the saga only needs to schedule/unschedule — so the base interface is enough.

### Practical guidance

For a typical saga schedule where the saga itself reacts to the scheduled message (expirations, retries, timeouts — the normal case), **always use `Schedule<TSaga, TMessage>`**. You need the `.Received` / `.AnyReceived` events in your behaviors, and those only exist on the two-parameter version.

Reach for `Schedule<TSaga>` only when writing reusable infrastructure code that shouldn't be coupled to a specific message type.

## Common errors and what they mean

| Error | Cause | Fix |
|---|---|---|
| `[Failure] BookReserved was not specified` | Event property declared + used in `When(...)` but no `Event(() => BookReserved, x => x.CorrelateBy...)` config. | Add `Event(() => BookReserved, x => x.CorrelateById(m => m.Message.ReservationId));`. |
| `[Failure] ExpirationSchedule.AnyReceived was not specified` | `Schedule<,>` declared/configured without `x.Received = r => r.CorrelateBy...`. | Supply the `x.Received` correlation inside the `Schedule(...)` lambda. |

## Infrastructure requirements

Scheduling needs a running scheduler. In the MassTransit Sample-Twitch infra, this is Quartz (`queue:quartz`) backed by SQL Server — both containers from `docker-compose.infra.yml` must be up. For tests, the in-memory scheduler / test harness handles this without external services.
